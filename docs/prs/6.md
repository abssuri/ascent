# PR: Issue 6 - Textual UI shell

Branch: `6-textual-ui-shell`. Source: `docs/plans/ascent.md`.

## Summary

Wire the headless core to a Textual terminal UI so the slice is playable end to
end. `AscentApp` owns the run and the current combat and switches between five
screens: Title, Map, Combat, Reward, and GameOver. Each screen renders a text
view of the core state and turns key presses into core commands; the core is
never mutated except through its own methods.

The flow: Title starts a new run and shows the Map. Entering a node begins a
combat (or heals and advances on a rest node). In combat the player spends energy
to play cards by number and ends the turn with `e`; winning shows the Reward
screen, dying shows GameOver. Taking the reward awards gold, advances the run,
and returns to the Map, or ends the run at GameOver once the boss is cleared.

Enemy content (a basic foe, an elite, and a boss) is added to `content.py` so a
node can be populated. `python3 -m ascent` launches the app; `--version` still
works and does not import Textual (the app import is deferred into `main`).

## How it was tested

One headless Pilot smoke test. It drives the app through `App.run_test()` with a
repeated block of keys (`1 2 3 4 5 e enter`) that advances whichever screen is
current, and asserts the run reaches the GameOver screen without raising. It
checks the screen machine holds together, not balance or layout. The existing
`test_layering.py` continues to guarantee the core never imports Textual.

## Cycle 2: GREEN

**Implementation**

- `content.py`: `basic_enemy` (Husk), `elite_enemy` (Warden), `boss_enemy`
  (The Gatekeeper), and `enemy_for(node_type)` to pick one per node.
- `src/ascent/ui/app.py`: `AscentApp` and the five screens. A shared
  `_TextScreen` base renders a single `Static` view of core state and refreshes
  it after each action. `AscentApp` owns the run and current combat and holds the
  transitions (`new_game`, `enter_node`, `go_reward`, `take_reward`,
  `go_game_over`). Combat card-play is guarded so an out-of-range or unaffordable
  index is a no-op rather than an error. Rest nodes heal 30 percent of max hp and
  advance. `_start_combat` builds a combat from the run's deck, shuffles the draw
  pile through the seeded `shuffle` stream, and draws the opening hand.
- `src/ascent/__main__.py`: parses `--seed` and launches `AscentApp`; the app
  import is deferred into `main` so `--version` never loads Textual.

**Rendering fix found by manual verification**

Driving the app headless and reading each screen's rendered text showed the
bracketed key hints (`[Enter]`, `[q]`, `[e]`) disappearing: Rich was parsing them
as markup tags. The view `Static` is now created with `markup=False`, so hint
text is literal. The smoke test would not have caught this (it asserts flow, not
text); the manual render check did.
