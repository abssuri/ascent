# PR: Issue 1 - Project scaffold and toolchain

Epic 0. Branch: `1-scaffold-project-tooling`. Source: `docs/plans/ascent.md`.

## Summary

Stand up the Python toolchain and package skeleton the rest of the project
depends on: a src-layout `ascent` package installed into a `.venv`, pinned dev
tooling (pytest, ruff, mypy) configured in a single `pyproject.toml`, headless
pygame test configuration via the SDL dummy drivers, and a `python3 -m ascent
--version` entry point. There is no game behaviour in this issue; it exists so
that every later issue has a working red/green harness and a green toolchain.

## How it was tested

The behavioural surface is one test asserting the command-line entry point. The
remaining acceptance criteria are toolchain gates, verified by the full `pytest`
suite, `ruff check`, `ruff format --check`, and `mypy` all exiting 0 on a clean
`.venv`. Those gate results are recorded in the GREEN cycle below once the entry
point is implemented.

## Cycle 1: RED

**Test surface**

- `tests/test_cli.py::test_version_flag_prints_version_and_exits_zero` runs
  `python -m ascent --version` in a subprocess and asserts exit code 0 and a
  version-shaped string on stdout.
- `tests/conftest.py` selects `SDL_VIDEODRIVER=dummy` and `SDL_AUDIODRIVER=dummy`
  before importing pygame, and a session fixture opens a 1x1 display surface so
  pygame runs windowless. Its correctness is demonstrated by the test suite
  collecting and running without a display.

**Coverage**

**RED evidence**

```
$ .venv/bin/python -m pytest
F                                                                        [100%]
FAILED tests/test_cli.py::test_version_flag_prints_version_and_exits_zero
  AssertionError: No module named ascent.__main__; 'ascent' is a package and
  cannot be directly executed
  assert 1 == 0
1 failed in 0.06s
```

The failure is an assertion on the missing entry point, not a collection or import
error, which confirms the harness and the headless pygame fixture work.

**Anticipated implementation order (GREEN)**

1. Add `src/ascent/__main__.py` with a `main()` that parses `--version`, prints
   the version from `ascent.__version__`, and exits 0.
2. Add a headless smoke test (`tests/test_headless.py`) asserting a pygame Surface
   can be created and blitted under the dummy driver.
3. Run and record the four gates green: `pytest`, `ruff check .`,
   `ruff format --check .`, `mypy`.

## Cycle 2: GREEN

**Implementation**

- `src/ascent/__main__.py` defines `main()`, the target of both `python -m ascent`
  and the `ascent` console script. It uses argparse's `version` action, which
  prints `ascent.__version__` to stdout and exits 0. The version string has a
  single source of truth in `ascent.__version__`.
- `tests/test_headless.py` adds the smoke test: it creates two `convert_alpha`
  surfaces, blits one onto the other under the dummy driver, and asserts the
  destination pixel matches the source colour. This exercises the parts of the
  session fixture (a real display surface for `convert_alpha`) that the CLI test
  does not.

**Lint cleanup**

The `# noqa: E402` directives on the pygame and pytest imports in `conftest.py`
were removed. The pinned ruff exempts imports that follow the `os.environ` driver
setup, so the directives were dead and tripped `RUF100`. The imports still sit
below the environment setup, which is the behaviour the fixture requires.
