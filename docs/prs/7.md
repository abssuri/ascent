# PR: Issue 7 - Horizontal combat layout

Branch: `7-horizontal-combat-ui`. Source: `docs/plans/ascent.md`.

## Summary

Re-lay the combat screen to match the horizontal mockup used when the UI was
chosen: a header with player hp, a boxed enemy panel showing hp, an hp bar, and
the telegraphed intent beside it, and the hand drawn as ASCII card boxes side by
side rather than a vertical list.

The layout is a pure `render_combat(combat)` function in
`src/ascent/ui/combat_view.py`. It imports only the core, holds no Textual
dependency, and returns the view as a monospace string. The `CombatScreen` calls
it, so the arrangement is unit-testable without a terminal.

## How it was tested

Two tests on `render_combat`: one asserts the cards render side by side (two card
names on the same line), the other asserts the header player hp, the enemy hp,
and the telegraphed intent all appear. Neither asserts exact spacing, so the
layout can be tuned without breaking them. The existing UI smoke test continues
to drive the whole flow through the reworked screen.

## Cycle 2: GREEN

**Implementation**

- `src/ascent/ui/combat_view.py`: `render_combat(combat)` and helpers (`_row`,
  `_hp_bar`, `_enemy_lines`, `_hand_lines`, `_card_column`, `_card_stat`). The
  header right-aligns player hp, the enemy is a box with an hp bar and the intent
  to its right, and the hand is built column by column and zipped so the boxes
  sit side by side. Card stat lines show cost then value, with block bracketed
  (`[5]`) and damage plain (`6`), matching the chosen mockup.
- `src/ascent/ui/app.py`: `CombatScreen.view_text` delegates to `render_combat`,
  and a `q` binding quits (the footer advertises it).
