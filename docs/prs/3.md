# PR: Issue 3 - Deterministic core: RNG and combat model

Branch: `3-deterministic-core-rng-and-combat`. Source: `docs/plans/ascent.md`.

## Summary

Lay the two deterministic foundations the combat engine builds on: a seeded RNG
group with named, independent streams that survive save/restore, and the
`Combatant` damage/block model. Both live in `ascent.core` and hold no reference
to the UI. This is the first game logic in the project.

`RngGroup(seed)` exposes named streams (`map`, `shuffle`, `enemy_ai`,
`card_reward`) seeded independently so consuming one never shifts another. Stream
seeds derive from sha256 of the seed and stream name, not the builtin `hash()`,
so a seed reproduces a run across separate processes. State serialises via
`Random.getstate()`/`setstate()`.

`Combatant` holds hp, max hp, and block. Damage is absorbed by block before hp,
block is spent as it absorbs, hp floors at zero, and block resets at the owner's
turn start.

## How it was tested

Eight tests across two files, kept deliberately small: five pin the RNG contract
(vocabulary, seed reproducibility, unknown-name rejection, stream independence,
save/restore), and three pin the combat model (block-before-hp, block reset, hp
floor and death). sha256-based seed derivation is a design choice (see Summary)
rather than a test-enforced one.
